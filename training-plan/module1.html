What's Git?

  • Keeps track of files in a project
  • Records any changes to project files
  • Restores previous versions of files
  • Compares and analyzes code
  • Merges code from different computers and different team members

Repositories?

  • Acts as a directory (folder) in your project's root directory
  • Local repos are on your computer
  • Remote repos serve as central hub for all team members

Git Initialization

  • git init
    • Git will initialize hidden .git directory for your project
  • git status
    • Shows any files in staging area, current branch, and tracked/untracked files

Tracking Files

  • git status
  • Information on files created, both tracked and untracked files
  • Files created locally are recognized by Git, but aren't in staging area yet
  • Must add files to staging area in order for Git to accept those file additions

git add

$ git add "file_name"

$ git rm --cached "file_name"

  • This command lets Git track files for a commit
  • Pass a filename with the command to let Git know what file to track
  • Can add multiple files at a time
    • git add . --> will add all files under current directory
  • git rm --cached my-file.ts --> removes files from staging area

git commit

$ git commit

  • Adding the -a character to git commit adds files to staging area too

$ git commit -a

git reset

$ git reset --soft HEAD^

$ git reset --hard HEAD^

  • Opposite of "add" command
  • --soft option cancels the commmit and moves before the HEAD
  • Allows for more files to be modified and added to staging area

Amending Commits

$ git add file-i-forgot-to-add.html

$ git commit --amend -m "Add the remaining file"

  • --amend lets you amend the last commit by adding a new file
  • You can also overwrite the message of your last commit

git branch

$ git branch -lavv

$ git branch new_branch

$ git checkout new_branch

$ git checkout -b new_branch_2

  • -lavv Lists the current working branch
  • Creates new branch (new_branch)
  • Creates new branch and switches to it (new_branch_2)
  • git checkout command is used to switch branches

git push/pull

$ git pull

# Insert changes to files...

$ git push origin devel

git merge

$ git checkout devel

$ git pull

$ git merge --no-ff feature_branch_to_merge

$ git branch -d feature_branch_to_merge

$ git push origin devel

  • First change into the branch that you want your changes to be added to
  • Merge changes from the feature branch into your devel/master branch
  • Will talk about specific workflows soon...

Fast Forward Merge

$ git merge --no-ff feature_branch_to_merge

  • Without --no-ff, there won't be any merge commit in the commit history

Comparing Workflows

  • Centralized
  • Feature Branch Workflow (Trunk Based)
  • Gitflow (what we use)
  • Centralized
    • Everything committed directly to master
    • Pros: Easy, good for small teams
    • Cons: Resolving conflicts can be a burden in larger teams, doesn't take advantage of git's distributive nature
  • Feature branch
    • Feature branches that are merged directly into master
    • Pros: Promotes team collaboration and code reviews, keeps master stable
    • Cons: Long lived branches higher risk of merge conflicts
  • Gitflow
    • Builds upon feature branch, has devel branch, release branch, etc
    • Pros: Ideal for projects on a release cycle, keeps master stable, releases made from stable master branch
    • Cons: High risk of merge conflicts

Centralized Overview

  • Only master branch is required
  • Similar to SVN, but each developer still has local copy of entire project
  • Access to Git's robust branching and merging model
    • Git branches are designed to for code integration and sharing changes
  • Works well for smaller teams

How It Works

  • First create central repo on a server
  • Each developer creates local copy of entire project
    • $ git clone ssh://user@host/path/to/repo.git
  • Commits are local, so developers can commit as much as they want without consequences in the central repo

Rebasing

$ git pull --rebase origin master
  • Before a developer can publish their feature, they need to fetch updated central commits and rebase their changes on top of them
  • Git recgonizes local changes/upstream commit conflicts
  • Rebasing is paused, conflicts are manually resolved, and rebase can complete.
$ git rebase --continue
  • Rebase procedure will continue until all local commits are checked and process is completed

Feature Branch Workflow (Trunk Based) Overview

  • Feature development takes place on separate branch than master
  • Allows multiple developers to work on a particular feature without disturbing main codebase
  • Branching model focused rather than repo focused

How It Works

  • Assumes a central repo, with its master branch representing "official" project history
  • Feature branches should be pushed to central repo once they are complete
  • Start with master branch, checkout a new feature branch to work on

How It Works Commands

$ git checkout master

$ git fetch origin

$ git reset --hard origin/master

$ git checkout -b new-feature

Gitflow Overview

  • Ideally suited for projects with a scheduled release cycle
  • No new concepts from feature branch workflow, each branch holds a specific role now
  • Uses individual branches for prepping, maintaining, and recording releases

How It Works

  • Workflow uses two branches, master and develop
  • Each new feature should reside in its own branch
  • Feature branches should use develop as their parent, and they should be merged back into develop

Bio::Neos Gitflow Commands

$ git checkout devel

$ git pull

$ git merge --no-ff feature_branch

$ git push origin devel

$ git branch -d feature_branch

Bio::Neos Gitflow Specs

  • Be sure to pull before attempting a merge.
  • If you ever get a rejected push after your merge, be sure to reset your devel branch to remove the incorporation of your branch by resetting to the branching commit, then starting over.
